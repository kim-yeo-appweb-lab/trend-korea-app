PRD 문서를 현재 코드베이스 상태와 동기화하여 최신화한다.

## 실행 방법

이 작업은 **Agent tool**을 사용하여 `prd-writer` 서브에이전트에게 위임한다.

- **subagent_type**: `prd-writer`
- **description**: "PRD 최신화"
- **prompt**: 이 커맨드의 "수행 절차" 이하 전체 내용을 전달한다. 기존 PRD 경로(`docs/PRD.md`)와 프로젝트 루트 경로를 포함한다.
- **mode**: "auto"

서브에이전트가 완료되면, 결과를 사용자에게 요약하여 보고한다.

## 수행 절차

### 1단계: 현황 파악

1. **PRD 읽기**: `docs/PRD.md`를 읽어 현재 요구사항을 파악한다.
2. **코드베이스 분석**: 실제 구현된 코드를 탐색하여 현재 상태를 파악한다.
   - 디렉토리 구조, 라우트, 주요 컴포넌트/모듈 확인
   - 도메인 모델(타입, 스키마, 엔티티) 확인
   - API 엔드포인트 확인
3. **ROADMAP 참조**: `docs/ROADMAP.md`가 있다면 완료/진행 상태를 참고한다.
4. **최근 변경 확인**: `git log --oneline -30`으로 최근 작업 방향을 파악한다.

### 2단계: 차이 분석

PRD와 코드베이스 사이의 차이를 다음 카테고리로 분류한다:

| 카테고리 | 설명 | 액션 |
|----------|------|------|
| **구현 완료** | PRD에 명시되어 있고, 코드에도 구현됨 | 상태 표기 추가/갱신 |
| **스펙 변경** | PRD 명세와 실제 구현이 다름 (의도적 변경) | PRD를 코드 기준으로 수정 |
| **미구현** | PRD에 있지만 아직 구현되지 않음 | 현행 유지 |
| **PRD 누락** | 코드에 있지만 PRD에 없음 (개발 중 추가된 기능) | PRD에 신규 항목 추가 |
| **도메인 모델 불일치** | 엔티티/필드명이 코드와 다름 | PRD를 코드 기준으로 수정 |

### 3단계: PRD 업데이트

차이 분석 결과를 바탕으로 PRD를 수정한다:

- **도메인 모델**: 실제 타입/스키마와 일치하도록 갱신
- **기능 명세**: 구현된 것과 다른 부분은 코드 기준으로 수정
- **API 스펙**: 실제 엔드포인트와 일치하도록 갱신
- **신규 항목**: 개발 중 추가된 기능은 적절한 섹션에 추가
- **변경 이력**: PRD 상단에 최신화 날짜와 변경 요약을 기록

### 4단계: 정합성 검증

- [ ] 도메인 모델이 실제 코드의 타입/스키마와 일치하는가?
- [ ] 삭제/변경된 기능이 PRD에서 정리되었는가?
- [ ] 새로 추가된 기능이 PRD에 반영되었는가?
- [ ] ROADMAP.md와 PRD 간 용어/범위가 일치하는가?

## 변경 이력 형식

PRD 상단 메타 영역에 다음을 추가/갱신한다:

```
> 최종 최신화: {날짜}
> 변경 사항: {한줄 요약}
```

## 주의사항

- `docs/PRD.md` 파일만 수정한다. 코드 파일은 절대 수정하지 않는다.
- 코드에 구현된 것이 PRD와 다를 때, **코드가 정본(Single Source of Truth)**이다.
- 의도적 변경인지 버그인지 판단이 어려운 경우, `[확인 필요]` 태그를 붙인다.
- 임의로 요구사항을 삭제하지 않는다. 폐기된 기능은 취소선(~~)으로 표기한다.
- 변경 사항을 카테고리별로 요약하여 사용자에게 보고한다.
