PRD 문서의 기술적 타당성을 검증한다.

## 실행 방법

이 작업은 **Agent tool**을 사용하여 `prd-technical-validator` 서브에이전트에게 위임한다.

- **subagent_type**: `prd-technical-validator`
- **description**: "PRD 기술 검증"
- **prompt**: 이 커맨드의 "수행 절차" 이하 전체 내용을 전달한다. PRD 경로(`docs/PRD.md`)와 프로젝트 루트 경로를 포함한다.
- **mode**: "auto"

서브에이전트가 완료되면, 결과를 사용자에게 요약하여 보고한다.

## 수행 절차

### 1단계: PRD 읽기

1. `docs/PRD.md`를 읽어 전체 요구사항을 파악한다.
2. 기술 스택, 아키텍처, 도메인 모델, API 스펙 등을 추출한다.

### 2단계: 기술적 타당성 검증

다음 관점에서 PRD를 검증한다:

| 검증 영역 | 검증 내용 |
|-----------|-----------|
| **기술적 모순** | 상충하는 요구사항이 없는가? |
| **구현 가능성** | 명시된 기술 스택으로 구현 가능한가? |
| **의존성 호환** | 라이브러리/프레임워크 간 버전 호환성이 있는가? |
| **누락된 엣지케이스** | 경계 조건, 에러 처리가 명시되어 있는가? |
| **모호한 요구사항** | 구현자가 해석에 따라 다르게 구현할 수 있는 부분이 있는가? |
| **보안 고려사항** | 인증/인가, 데이터 보호 관련 누락이 있는가? |
| **성능 요구사항** | 비현실적이거나 누락된 성능 기준이 있는가? |

### 3단계: 코드베이스 대조 (선택적)

코드베이스가 존재하는 경우:

1. 실제 구현과 PRD 스펙 간의 불일치를 식별한다.
2. 구현되었으나 PRD에 없는 기능을 식별한다.
3. PRD에 명시되었으나 기술적으로 현 아키텍처에서 구현이 어려운 항목을 식별한다.

### 4단계: 검증 보고서 작성

검증 결과를 다음 형식으로 정리하여 사용자에게 보고한다:

```
## 검증 결과 요약

### 🔴 심각 (구현 차단)
- [항목]: [설명]

### 🟡 주의 (구현 가능하나 리스크)
- [항목]: [설명]

### 🟢 권장 (개선 사항)
- [항목]: [설명]

### ✅ 검증 통과
- [항목 목록]
```

## 주의사항

- PRD 파일은 수정하지 않는다. 검증 결과만 보고한다.
- 코드에 대한 수정도 하지 않는다.
- 판단이 어려운 사항은 `[확인 필요]` 태그를 붙인다.
- 검증 기준은 프로젝트의 기술 스택과 규모에 맞게 적용한다.
