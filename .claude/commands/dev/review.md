현재 변경사항에 대해 코드 리뷰를 수행한다.

## 수행 절차

### 1단계: 변경사항 파악

1. `git diff`로 스테이징되지 않은 변경사항을 확인한다.
2. `git diff --cached`로 스테이징된 변경사항을 확인한다.
3. 변경된 파일 목록과 변경 범위를 파악한다.

### 2단계: 컨벤션 체크

`.claude/rules/` 하위의 **모든 규칙 파일**을 읽고, 변경된 파일의 `paths` 패턴과 매칭되는 규칙을 적용하여 검증한다.

각 규칙 파일의 frontmatter `paths` 필드를 기준으로:
- 변경된 `.ts`/`.tsx` 파일 → `code-conventions.md` 적용
- 변경된 `.tsx` 파일 → `ui.md` 추가 적용
- 변경된 테스트 파일 → `testing.md` 적용

### 3단계: 품질 검증

- **보안**: 하드코딩된 시크릿, XSS, 인젝션 취약점
- **접근성**: `a11y.md` 규칙 기반 검증 (UI 변경 시)
- **타입 안전성**: 타입 단언 남용, 불필요한 타입 캐스팅
- **에러 처리**: 누락된 에러 핸들링, 사용자에게 노출되는 에러 메시지

### 4단계: 리뷰 결과 보고

다음 형식으로 결과를 정리한다:

| 파일 | 라인 | 심각도 | 내용 |
|------|------|--------|------|
| ... | ... | 🔴/🟡/🟢 | ... |

- 🔴 **필수 수정**: 버그, 보안 이슈, 컨벤션 위반
- 🟡 **권장 수정**: 가독성, 유지보수성 개선
- 🟢 **참고**: 스타일 제안, 대안 접근법

## 주의사항

- 코드를 수정하지 않는다. 리뷰 결과만 보고한다.
- 변경되지 않은 코드에 대한 지적은 하지 않는다.
- 이슈가 없으면 간단히 "이슈 없음"으로 보고한다.

## 관련 커맨드

- `/git:commit` — 리뷰 통과 후 커밋
- `/dev:feature` — 기능 개발로 돌아가기
